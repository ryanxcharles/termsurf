# Issue 318: Cmd Keybindings for Webview

## Goal

Complete the clipboard keybindings (Cmd+C/V/X/A) for browser interaction in ts3
webviews. This is a focused continuation of issue 317 (input forwarding).

## Status

| Keybinding         | Status      | Notes                                     |
| ------------------ | ----------- | ----------------------------------------- |
| Cmd+V (paste)      | Done        | Issue 317, exp 6 (JS injection)           |
| Cmd+C (copy)       | Done        | Experiment 1 (`frame.copy()` via XPC)     |
| Cmd+X (cut)        | Done        | Experiment 2 (`frame.cut()` via XPC)      |
| Cmd+A (select all) | Not started | Use CEF's native `frame.select_all()` API |

## Proposed Experiments

| #   | Feature          | Approach                                                   |
| --- | ---------------- | ---------------------------------------------------------- |
| 1   | Cmd+C (copy)     | GUI sends `do_copy` via XPC, profile calls `frame.copy()`  |
| 2   | Cmd+X (cut)      | GUI sends `do_cut` via XPC, profile calls `frame.cut()`    |
| 3   | Cmd+A (select)   | GUI sends `do_select_all`, profile calls `frame.select_all()` |

## Background

### What Works (from Issue 317)

1. **Basic keyboard input** (experiment 1): Typing, arrows, Tab, Enter,
   Backspace
2. **Skip keybindings in Browse mode** (experiment 3): WezTerm keybindings don't
   fire when webview has focus
3. **KeyEvent generation for Cmd shortcuts** (experiment 5): Modified
   `performKeyEquivalent:` to synthesize KeyEvents for Cmd+C/V/X/A
4. **Clipboard paste via XPC** (experiment 6): GUI reads clipboard, sends via
   XPC, profile server injects via JavaScript

### Architecture Constraints

**Paste requires proxying**: The profile server cannot READ the system clipboard
because it's not the active application. Paste must be proxied through the GUI.

**Copy/Cut/Select may work natively**: WRITING to the clipboard may not have the
same restriction. CEF's `Frame` provides direct methods (`copy()`, `cut()`,
`select_all()`) that should work without proxying.

```
Paste (Cmd+V) - requires proxy:
[GUI]  Read clipboard → send text via XPC → [Profile] inject via JS

Copy/Cut/Select (Cmd+C/X/A) - native CEF:
[GUI]  Send command via XPC → [Profile] call frame.copy()/cut()/select_all()
```

### Key Files

| File                                            | Role                                             |
| ----------------------------------------------- | ------------------------------------------------ |
| `ts3/window/src/os/macos/window.rs`             | `performKeyEquivalent:` intercepts Cmd shortcuts |
| `ts3/wezterm-gui/src/termwindow/keyevent.rs`    | Detects Cmd+V, reads clipboard, calls XPC        |
| `ts3/wezterm-gui/src/termwindow/webview_xpc.rs` | XPC methods for clipboard operations             |
| `ts3/termsurf-profile/src/main.rs`              | Handles XPC commands, JavaScript injection       |

## Experiments

### Experiment 1: Cmd+C (Copy)

**Goal:** Copy selected text from the browser to the system clipboard.

#### Approach

CEF's `Frame` has a native `copy()` method that copies the current selection to
the system clipboard. Unlike paste (which requires reading the clipboard), copy
only writes — this may work even from a background process.

**Flow:**

1. User presses Cmd+C in Browse mode
2. GUI sends XPC message `{ action: "do_copy" }` to profile server
3. Profile server calls `frame.copy()` on the CEF UI thread
4. CEF handles clipboard write internally

**Part A: Add send_copy to XpcManager (webview_xpc.rs)**

```rust
/// Send copy command to the browser
pub fn send_copy(&self, pane_id: PaneId) -> bool {
    let msg = XpcDictionary::new();
    msg.set_string("action", "do_copy");
    self.send_command(pane_id, &msg)
}
```

**Part B: Handle do_copy in profile server (main.rs)**

Add alongside the existing `paste_text` handler:

```rust
"do_copy" => {
    let state_guard = deferred_for_handler.lock().unwrap();
    let Some(bs) = state_guard.as_ref() else {
        println!("Profile: do_copy ignored (state not ready)");
        return;
    };

    let bs = Arc::clone(bs);
    drop(state_guard);

    // Post to CEF UI thread
    let mut task = CopyTask::new(bs);
    cef::post_task(cef::ThreadId::UI, Some(&mut task));
}
```

**Part C: CopyTask implementation (main.rs)**

```rust
wrap_task! {
    pub struct CopyTask {
        state: Arc<BrowserState>,
    }

    impl Task {
        fn execute(&self) {
            if let Some(browser) = self.state.browser.lock().unwrap().as_ref() {
                if let Some(frame) = browser.main_frame() {
                    println!("[CLIPBOARD] Calling frame.copy()");
                    frame.copy();
                }
            }
        }
    }
}
```

**Part D: Detect Cmd+C in GUI (keyevent.rs)**

In `handle_webview_key_event`, Browse mode section:

```rust
// Handle Cmd+C (copy) - call CEF's native copy
let is_cmd_c = window_key.key_is_down
    && window_key.modifiers.contains(Modifiers::SUPER)
    && matches!(&window_key.key, KeyCode::Char('c') | KeyCode::Char('C'));

if is_cmd_c {
    if let Some(xpc_manager) = crate::termwindow::webview_xpc::get_xpc_manager() {
        log::info!("[CLIPBOARD] Cmd+C detected, sending do_copy to browser");
        xpc_manager.send_copy(pane_id);
    }
    return Some(true);
}
```

#### Files to Modify

| File                                            | Changes                    |
| ----------------------------------------------- | -------------------------- |
| `ts3/wezterm-gui/src/termwindow/keyevent.rs`    | Detect Cmd+C, call send_copy |
| `ts3/wezterm-gui/src/termwindow/webview_xpc.rs` | Add `send_copy()` method   |
| `ts3/termsurf-profile/src/main.rs`              | Handle `do_copy`, call `frame.copy()` |

#### Verification

```bash
cd ts3 && ./scripts/build-debug.sh --open

# Test 1: Copy text from browser
web google.com
# Select text with Shift+arrows
# Press Cmd+C
# Paste in another app (e.g., Notes, TextEdit)
# Expected: Selected text was copied

# Test 2: Copy from input field
# Type text in search box
# Select with Shift+arrows
# Press Cmd+C
# Paste elsewhere
# Expected: Selected text was copied

# Check logs
cat /tmp/termsurf-gui.log | grep "CLIPBOARD"
cat /tmp/termsurf-profile-*.log | grep "CLIPBOARD"
```

#### Success Criteria

- [x] Cmd+C detected in Browse mode
- [x] `do_copy` XPC message sent to profile server
- [x] Profile server calls `frame.copy()` on UI thread
- [x] Selected text appears in system clipboard
- [x] Can paste copied text in other apps

#### Result: Success

CEF's native `frame.copy()` works from the background profile server process.
The clipboard write operation is not restricted like clipboard read (paste).
Copy works for both page content and input field selections.

#### Conclusion

The asymmetry hypothesis was correct: macOS restricts clipboard *reading* for
background processes (security concern), but allows clipboard *writing*. This
means:

- **Paste (Cmd+V)**: Must proxy through GUI (read clipboard → XPC → JS inject)
- **Copy (Cmd+C)**: Direct CEF API works (`frame.copy()` writes to clipboard)

This pattern should apply to Cmd+X (cut) and Cmd+A (select all) as well, since
`frame.cut()` writes to clipboard and `frame.select_all()` doesn't touch the
clipboard at all.

### Experiment 2: Cmd+X (Cut)

**Goal:** Cut selected text from the browser to the system clipboard.

#### Approach

Identical to experiment 1, but using `frame.cut()` instead of `frame.copy()`.
Cut copies the selection to clipboard AND deletes it from the document.

**Flow:**

1. User presses Cmd+X in Browse mode
2. GUI sends XPC message `{ action: "do_cut" }` to profile server
3. Profile server calls `frame.cut()` on the CEF UI thread
4. CEF copies selection to clipboard and deletes it

**Part A: Add send_cut to XpcManager (webview_xpc.rs)**

```rust
/// Send cut command to the browser (issue 318, experiment 2)
pub fn send_cut(&self, pane_id: PaneId) -> bool {
    let msg = XpcDictionary::new();
    msg.set_string("action", "do_cut");
    self.send_command(pane_id, &msg)
}
```

**Part B: Handle do_cut in profile server (main.rs)**

Add alongside the existing `do_copy` handler:

```rust
"do_cut" => {
    let state_guard = deferred_for_handler.lock().unwrap();
    let Some(bs) = state_guard.as_ref() else {
        println!("Profile: do_cut ignored (state not ready)");
        return;
    };

    println!("[CLIPBOARD] Received do_cut command");

    let bs = Arc::clone(bs);
    drop(state_guard);

    // Post to CEF UI thread
    let mut task = CutTask::new(bs);
    cef::post_task(cef::ThreadId::UI, Some(&mut task));
}
```

**Part C: CutTask implementation (main.rs)**

```rust
wrap_task! {
    pub struct CutTask {
        state: Arc<BrowserState>,
    }

    impl Task {
        fn execute(&self) {
            if let Some(browser) = self.state.browser.lock().unwrap().as_ref() {
                if let Some(frame) = browser.main_frame() {
                    println!("[CLIPBOARD] Calling frame.cut()");
                    frame.cut();
                }
            }
        }
    }
}
```

**Part D: Detect Cmd+X in GUI (keyevent.rs)**

In `handle_webview_key_event`, Browse mode section (after Cmd+C handler):

```rust
// Handle Cmd+X (cut) - call CEF's native cut (issue 318, experiment 2)
let is_cmd_x = window_key.key_is_down
    && window_key.modifiers.contains(Modifiers::SUPER)
    && matches!(&window_key.key, KeyCode::Char('x') | KeyCode::Char('X'));

if is_cmd_x {
    log::info!("[CLIPBOARD] Cmd+X detected, sending do_cut to browser");
    drop(overlays);
    if let Some(xpc_manager) = crate::termwindow::webview_xpc::get_xpc_manager() {
        xpc_manager.send_cut(pane_id);
    }
    return Some(true);
}
```

#### Result: Success

CEF's native `frame.cut()` works identically to `frame.copy()`. The selection
is copied to the clipboard and deleted from the document in a single operation.

#### Conclusion

No new architectural insights — experiment 2 confirms the pattern from
experiment 1. The CEF `Frame` clipboard methods (`copy()`, `cut()`) work
reliably from a background process. Implementation was mechanical: same XPC
message pattern, same task structure, just a different CEF method call.

This validates the "one experiment at a time" approach: experiment 1 established
the pattern, experiment 2 was pure replication with minimal risk.

## Future Work

### Phase 2: Mouse Input (Issue TBD)

After completing keyboard input, mouse input is the next major feature:

| Feature      | Description                                      |
| ------------ | ------------------------------------------------ |
| Mouse click  | Forward left/right/middle click to CEF           |
| Mouse move   | Forward hover events for hover effects, tooltips |
| Mouse drag   | Text selection via drag                          |
| Scroll wheel | Page scrolling                                   |
| Double-click | Word selection                                   |
| Triple-click | Line/paragraph selection                         |

**Architecture:**

- Intercept mouse events in WezTerm's event handler
- Hit-test to determine if mouse is over webview pane
- Translate coordinates from physical window pixels to logical browser
  coordinates
- Send via XPC to profile server
- Profile server calls CEF's mouse event APIs

### Ctrl+C Context Sensitivity

Currently, Ctrl+C always switches to Control mode. A more sophisticated
approach:

1. Query CEF for current selection state
2. If text is selected: Ctrl+C copies (like Cmd+C)
3. If no selection: Ctrl+C switches to Control mode

This requires querying CEF for selection state, which may need JavaScript or
CEF's selection APIs.
