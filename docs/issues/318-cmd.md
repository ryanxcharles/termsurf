Complete the clipboard keybindings (Cmd+C/V/X/A) for browser interaction in ts3
webviews. This is a focused continuation of issue 317 (input forwarding).

## Background

### What Works (from Issue 317)

1. **Basic keyboard input** (experiment 1): Typing, arrows, Tab, Enter,
   Backspace
2. **Skip keybindings in Browse mode** (experiment 3): WezTerm keybindings don't
   fire when webview has focus
3. **KeyEvent generation for Cmd shortcuts** (experiment 5): Modified
   `performKeyEquivalent:` to synthesize KeyEvents for Cmd+C/V/X/A
4. **Clipboard paste via XPC** (experiment 6): GUI reads clipboard, sends via
   XPC, profile server injects via JavaScript

### Architecture Constraints

**Process clipboard isolation**: The profile server cannot access the system
clipboard because it's not the active application. All clipboard operations must
be proxied through the GUI process via XPC:

```
[GUI Process]              [Profile Server Process]
     │                              │
     │  1. User presses Cmd+V       │
     ▼                              │
Read system clipboard               │
     │                              │
     │  2. Send text via XPC        │
     ├─────────────────────────────►│
     │                              ▼
     │                   Inject via JavaScript
     │                   document.execCommand('insertText')
     │                              │
     │  3. User presses Cmd+C       │
     │                              ▼
     │                   Query via JavaScript
     │                   window.getSelection().toString()
     │                              │
     │  4. Send text via XPC        │
     │◄─────────────────────────────┤
     ▼                              │
Write to system clipboard           │
```

## Experiments

### Experiment 1: Cmd+C (Copy)

**Goal:** Copy selected text from the browser to the system clipboard.

#### Approach

**Flow:**

1. User presses Cmd+C in Browse mode
2. GUI sends XPC message `{ action: "get_selection" }` to profile server
3. Profile server executes JavaScript to query selection
4. Profile server sends selection back to GUI via XPC response
5. GUI writes selection to system clipboard

**Part A: Add XPC request/response for selection (webview_xpc.rs)**

The existing XPC architecture is fire-and-forget. We need a response channel.

Option 1: Synchronous XPC request with
`xpc_connection_send_message_with_reply_sync` Option 2: Callback-based with
state tracking

For simplicity, use Option 1 with a short timeout:

```rust
/// Request selected text from the browser
pub fn get_selection(&self, pane_id: PaneId) -> Option<String> {
    let msg = XpcDictionary::new();
    msg.set_string("action", "get_selection");

    // Send synchronous request with timeout
    let response = self.send_request_with_reply(pane_id, &msg, Duration::from_millis(500))?;
    response.get_string("text").map(|s| s.to_string())
}
```

**Part B: Handle get_selection in profile server (main.rs)**

```rust
"get_selection" => {
    let state_guard = deferred_for_handler.lock().unwrap();
    let Some(bs) = state_guard.as_ref() else {
        // Return empty response
        return;
    };

    let bs = Arc::clone(bs);
    drop(state_guard);

    // Execute JavaScript to get selection
    let mut task = GetSelectionTask::new(bs, reply_callback);
    cef::post_task(cef::ThreadId::UI, Some(&mut task));
}
```

**Challenge:** CEF's `execute_java_script` is fire-and-forget. Getting the
result requires one of:

1. **CEF message router**: Register a JavaScript handler that calls back to
   native
2. **Custom scheme handler**: Inject JS that fetches
   `termsurf://selection?text=...`
3. **DOM query via DevTools protocol**: Use CDP to query the DOM

For simplicity, use approach 2 with a custom scheme:

```javascript
// Injected JavaScript
(function() {
    var sel = window.getSelection().toString();
    fetch('termsurf://clipboard/copy?text=' + encodeURIComponent(sel));
})();
```

Then handle the custom URL in a request handler.

#### Alternative: Simpler Keyboard-Only Approach

Since the complexity of bidirectional communication is high, consider:

1. For Cmd+C: Forward the key event to CEF and let the browser's native copy
   work
2. Problem: CEF's off-screen browser may not have clipboard access either

This needs investigation. The JavaScript approach is the reliable fallback.

## Future Work

### Phase 2: Mouse Input (Issue TBD)

After completing keyboard input, mouse input is the next major feature:

| Feature      | Description                                      |
| ------------ | ------------------------------------------------ |
| Mouse click  | Forward left/right/middle click to CEF           |
| Mouse move   | Forward hover events for hover effects, tooltips |
| Mouse drag   | Text selection via drag                          |
| Scroll wheel | Page scrolling                                   |
| Double-click | Word selection                                   |
| Triple-click | Line/paragraph selection                         |

**Architecture:**

- Intercept mouse events in WezTerm's event handler
- Hit-test to determine if mouse is over webview pane
- Translate coordinates from physical window pixels to logical browser
  coordinates
- Send via XPC to profile server
- Profile server calls CEF's mouse event APIs

### Ctrl+C Context Sensitivity

Currently, Ctrl+C always switches to Control mode. A more sophisticated
approach:

1. Query CEF for current selection state
2. If text is selected: Ctrl+C copies (like Cmd+C)
3. If no selection: Ctrl+C switches to Control mode

This requires the same bidirectional XPC infrastructure as Cmd+C.
