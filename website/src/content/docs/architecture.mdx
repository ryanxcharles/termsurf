<h1>Architecture</h1>

<p>
  TermSurf is a protocol, not just an app. It is a network of interchangeable
  components — terminals, browser engines, and TUIs — all speaking the same
  protobuf/Unix socket protocol.
</p>

<h2>Multi-process</h2>

<p>
  TermSurf is multi-process by necessity. Each browser engine process serves
  exactly one profile (one set of cookies, storage, and cache). This is a hard
  constraint imposed by Chromium (one <code>BrowserContext</code> per process),
  and Gecko and Ladybird have the same limitation.
</p>

<p>
  The multi-process design enables multi-engine support. Because each browser
  process is an independent program speaking the TermSurf protocol, the GUI
  doesn't care which engine is behind it. A user can have one pane running
  Roamium (Chromium), another running Surfari (WebKit), and a third running
  Girlbat (Ladybird) — all in the same terminal window.
</p>

<h2>Process Topology</h2>

<p>
  N TUIs connect to 1 GUI. The GUI manages M browser engine processes, each
  serving one profile. Different profiles can use different engines.
</p>

<p>
  The GUI is the hub — it routes messages between TUIs and engines, manages pane
  layout, and composites browser overlays into the terminal window.
</p>

<h2>Unix Sockets + Protobuf</h2>

<p>
  All inter-process communication uses Unix domain sockets with length-prefixed
  protobuf messages.
</p>

<ul>
  <li>
    <strong>TUI &rarr; GUI:</strong> The TUI reads the
    <code>TERMSURF_SOCKET</code> env var (set by the GUI) to discover the socket
    path.
  </li>
  <li>
    <strong>GUI &rarr; Engine:</strong> The GUI passes
    <code>--ipc-socket=&#123;path&#125;</code> when launching browser engine
    processes.
  </li>
  <li>
    <strong>Wire format:</strong> 4-byte little-endian length prefix +
    serialized protobuf.
  </li>
  <li>
    <strong>Serialization:</strong> prost in Rust (GUI, TUI, engines), C++
    protobuf in Chromium.
  </li>
</ul>

<h2>GPU Compositing</h2>

<p>
  On macOS, browser content is rendered by Chromium's GPU process and composited
  into the terminal window via <code>CALayerHost</code>. This is zero-copy GPU
  rendering — the browser's framebuffer is shared with the terminal's window
  server through a <code>CAContext</code> layer ID. No pixel copying, no shared
  memory buffers.
</p>

<h2>Browser Engines</h2>

<p>
  Each engine follows the same pattern: a C shared library wrapping the engine's
  embedding API (<code>ts_*</code> functions), linked by a Rust binary that
  handles Unix socket IPC, protobuf parsing, and process lifecycle. The Rust
  binary (~400 lines) is almost entirely reusable across engines.
</p>

<h2>Multiple GUIs</h2>

<p>
  Any terminal emulator that implements the TermSurf protocol can host browser
  overlays. Current and planned GUIs:
</p>

<ul>
  <li>
    <strong>Ghostboard</strong> — Active. Ghostty fork, Zig/Swift.
  </li>
  <li>
    <strong>Wezboard</strong> — Archived in git history. Former WezTerm fork,
    Rust.
  </li>
  <li>
    <strong>Kitty</strong> — Planned.
  </li>
  <li>
    <strong>Alacritty</strong> — Planned.
  </li>
  <li>
    <strong>iTerm2</strong> — Planned.
  </li>
</ul>

<h2>Many TUIs</h2>

<p>
  The first TUI, <code>web</code>, provides browser chrome (URL bar, navigation,
  modes) in the terminal pane. But TermSurf is a webview overlay protocol — many
  TUIs can embed web browsers:
</p>

<ul>
  <li>
    <code>web</code> — general-purpose web browser TUI (current)
  </li>
  <li>
    Future: documentation viewers, API explorers, email clients, dashboard
    monitors
  </li>
</ul>
