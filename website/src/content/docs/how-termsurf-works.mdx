<h1>How TermSurf Works</h1>

<p>
  TermSurf puts a real web browser inside a terminal pane. You type
  <code>web example.com</code> and the page renders right where you're working —
  no alt+tab, no switching to a separate browser app, no losing your place. You
  drive it from the keyboard with a small chrome (a URL bar and status line) and
  a couple of modes.
</p>

<h2>What happens when you run <code>web example.com</code></h2>

<ol>
  <li>
    The <a href="/docs/components/webtui"><code>web</code> TUI</a> starts in the
    pane and connects to the GUI (Ghostboard) over a Unix socket (it finds the
    socket via the <code>TERMSURF_SOCKET</code> environment variable the GUI
    sets).
  </li>
  <li>
    The GUI launches — or reuses — a browser engine process for your
    <strong>profile</strong>. There is one engine process per profile, so each
    profile keeps its own cookies, storage, and cache.
  </li>
  <li>
    The engine renders the page, and the GUI composites it as a
    <strong>zero-copy GPU overlay</strong> positioned exactly over your pane (on
    macOS, via <code>CALayerHost</code>). The web content is drawn by the GPU,
    not painted as terminal cells.
  </li>
  <li>
    The TUI draws the chrome (URL bar, status) in the terminal and forwards your
    input. You switch between <strong>Control</strong> mode (terminal/TUI keys)
    and <strong>Browse</strong> mode (keyboard and mouse go to the page).
  </li>
</ol>

<p>
  The browser lives in the pane alongside your shells and editors — resize it,
  split it, or close it like any other pane. To move between editing the URL and
  using the page, you change modes rather than switching apps; see the
  <a href="/docs/components/webtui">Web TUI</a> reference for the modes,
  keybindings, and commands.
</p>

<h2>Why it's built this way</h2>

<p>
  TermSurf is a protocol, not just an app: the terminal (GUI), each browser
  engine, and each TUI are separate processes speaking one protobuf/Unix-socket
  protocol. That separation is what makes the engine swappable and the overlay
  model possible. The full process topology, the IPC wire format, and the
  GPU-compositing mechanics are documented in
  <a href="/docs/architecture">Architecture</a>; the message types are in the
  <a href="/docs/protocol/overview">protocol overview</a>.
</p>

<p>
  Today the shipping engine is <a href="/docs/components/roamium">Roamium</a>
  (Chromium) on macOS. Because the GUI talks to engines only through the
  protocol, additional engines are a planned extension rather than a rewrite —
  see the engine table in <a href="/docs/architecture">Architecture</a>.
</p>
