# Experiment 152: Phase I — generated width table

## Description

Experiment 151 proved the terminal print path can use a dedicated width-only API
without weakening Unicode metadata behavior, but its scalar range helper was
slower than the existing generated `Properties` lookup. This experiment keeps
the same width-only API and changes only the backing strategy: generate a
compact width-only stage table from Ghostty's pinned Unicode data and use that
for `unicode::codepoint_width`.

The target is not a new width oracle. The generated `Properties` table remains
the source of truth for grapheme-break, variation-selector, and
`width_zero_in_grapheme` metadata, and the width-only table must be derived from
the same `stage3` entries as `STAGE3`. The goal is to make the terminal hot path
load one byte instead of copying a full `Properties` value while preserving the
same staged index structure and the same exhaustive scalar parity guarantees.

This experiment closes the Phase I SIMD/perf width item only if the release perf
probe demonstrates at least a 1.05x speedup over `table_properties(codepoint)`
width lookup. If the generated table is correct but not faster, the result stays
Partial and the remaining work moves to target intrinsics or a C++ Highway
bridge.

## Changes

- `scripts/roastty-app/generate-unicode-tables.py`
  - Extend `render_props()` so the generated Rust table also emits a compact
    width-only `WIDTH_STAGE3` array derived directly from each parsed
    `Props.width`.
  - Keep `WIDTH_STAGE3` regenerated by both `--generate` and `--check`, so any
    future Ghostty Unicode-data drift updates the full property table and
    width-only table together.
- `roastty/src/unicode/tables.rs`
  - Regenerate the file from the script; do not hand-edit the generated table.
  - Add the generated `pub(crate) const WIDTH_STAGE3: [u8; N]` beside `STAGE3`.
- `roastty/src/unicode/mod.rs`
  - Replace the scalar range helper fallback chain with a width-only staged
    lookup that reads `WIDTH_STAGE3[stage3_index]`.
  - Preserve the documented caller-filter semantics from Experiment 151:
    printable/non-control codepoints `<= 0xff` return width `1`;
    `TerminalStreamHandler::print` must still reject C0/C1 controls before the
    `<= 0xff` fast path can widen them; out-of-range codepoints still return
    Ghostty's fallback width `1`.
  - Keep `unicode::get` and `table_properties` unchanged for metadata-heavy
    callers.
  - Keep the exhaustive valid-scalar parity test, now proving that the generated
    width table matches `table_properties(...).width` for every valid Unicode
    scalar value with the same `<= 0xff` control exception.
  - Adjust the ignored release perf probe so it compares the new generated
    width-only lookup against direct full-property lookup and fails below a
    1.05x speedup.
- `roastty/src/unicode/width.rs`
  - Remove the scalar range helper if no longer used, or reduce it to a tiny
    wrapper only if the implementation keeps the width table in a separate
    module.
- `roastty/src/terminal/terminal.rs`
  - No behavioral change is intended beyond using the existing
    `unicode::codepoint_width` API; keep the C1-control regression test from
    Experiment 151.
- `issues/0802-libroastty-completion-and-mac-app/README.md`
  - If width parity and perf pass, mark the Phase I SIMD checklist item complete
    and record that generated width tables close width without a C++ bridge.
  - If perf remains below 1.05x, keep the checklist open and record the exact
    next required intrinsics/bridge work.

## Verification

- `scripts/roastty-app/generate-unicode-tables.py --check`
- `cargo fmt`
- `cargo test -p roastty unicode -- --test-threads=1`
- `cargo test -p roastty terminal_stream -- --test-threads=1`
- `cargo test -p roastty -- --test-threads=1`
- `cd roastty && macos/build.nu --action test`
- `cargo fmt --check`
- `git diff --check`
- `cargo test --release -p roastty simd_fast_path_perf_codepoint_width -- --ignored --nocapture --test-threads=1`

**Pass** = the generated width-only table is produced from the pinned Ghostty
Unicode data, `unicode::codepoint_width` uses it without changing terminal
print/grapheme behavior, exhaustive scalar parity passes, the release perf probe
shows at least a 1.05x speedup, all listed Rust and hosted macOS checks pass,
and the README marks the Phase I SIMD checklist item complete.

**Partial** = the generated width table is correct and wired but the release
perf probe is below 1.05x, or the implementation must keep enough full-property
lookup work in the hot path that the SIMD/perf roadmap item remains open.

**Fail** = the generated width table diverges from `STAGE3`, changes terminal
print or grapheme behavior, cannot be regenerated reproducibly, or cannot be
verified by the listed checks.
