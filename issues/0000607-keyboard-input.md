# Issue 607: Keyboard Input

## Goal

Type in text fields, search boxes, and form inputs on Chromium overlays. Support
Cmd+C to copy selected text, Cmd+V to paste, Cmd+A to select all, Tab to move
between form fields, Enter to submit, Backspace to delete, and arrow keys to
navigate within text. Escape to exit browse mode.

## Background

Issue 606 completed the mouse input pipeline: clicking, scrolling, text
selection, cursor sync, focus lifecycle, and activation gating — all in Zig. But
there is no keyboard input. Typing in a search box or text field does nothing.
Cmd+C doesn't copy selected text.

Ghost's keyboard pipeline is:

1. macOS `keyDown`/`keyUp` in `SurfaceView_AppKit.swift`
2. `NSTextInputClient` protocol processes the event (dead keys, IME, etc.)
3. `ghostty_surface_key` calls into Zig
4. `Surface.keyCallback()` receives `input.KeyEvent` with:
   - `action`: press, release, repeat
   - `key`: physical key enum (layout-agnostic, e.g., `.a`, `.enter`, `.tab`)
   - `mods`: shift, ctrl, alt, super
   - `utf8`: text generated by the keystroke (e.g., "a", "A", "ñ")
   - `composing`: true during dead key / IME composition
   - `unshifted_codepoint`: the unshifted character (e.g., 'a' for Shift+A)

Chromium's `ForwardKeyboardEvent` takes a `NativeWebKeyboardEvent` which wraps
`blink::WebKeyboardEvent`. The key fields are:

- `type`: `kRawKeyDown`, `kKeyUp`, `kChar`
- `windows_key_code`: Windows VK code (e.g., `0x41` for 'A', `0x0D` for Enter)
- `dom_code`: USB HID usage code
- `dom_key`: DOM key enum value
- `text[4]`: UTF-16 text generated by the key
- `unmodified_text[4]`: UTF-16 text without modifiers (except shift)
- `modifiers`: bitmask (shift=1, ctrl=2, alt=4, meta=8 — same as mouse)

The modifier bitmask matches what we already send for mouse events.

### Chromium keyboard event protocol

For a typical key press (e.g., typing "a"), Chromium expects two events:

1. `kRawKeyDown` with `windows_key_code` set, `text` empty
2. `kChar` with `windows_key_code` set to the character code, `text` set

For modifier-only keys (Shift, Ctrl, etc.) or non-character keys (Arrow, Escape,
F1, etc.), only `kRawKeyDown` is sent (no `kChar`).

For key release, `kKeyUp` is sent.

### What we need on the Chromium side

The Chromium Profile Server currently has no keyboard event handler. We need to
add:

1. `HandleKeyEvent` method on `ShellBrowserMainParts` (similar to
   `HandleMouseEvent`)
2. XPC message handler for `"key_event"` action
3. Construct `NativeWebKeyboardEvent` from the XPC fields
4. Call `ForwardKeyboardEvent` on the `RenderWidgetHost`

### Key mapping

Ghostty's `input.Key` enum uses DOM-style names (`.a`, `.enter`, `.tab`,
`.arrow_up`). These need to map to Windows virtual key codes (`VK_*`) for
Chromium's `windows_key_code` field. The mapping is well-defined — Chromium's
`ui/events/keycodes/keyboard_codes_posix.h` has all the constants.

### Scope

This issue focuses on getting basic keyboard input working end-to-end. Advanced
topics like IME composition, dead keys, and international layouts are deferred
to a later issue.

## Experiment 1: Ctrl+Esc exits browse mode

### Goal

Pressing Ctrl+Esc while in browse mode immediately exits to control mode. This
is a reliable escape hatch that works regardless of browser state.

### Background

Once keyboard input forwarding is implemented, the Escape key will have meaning
inside the browser — exit an input box, close a dialog, dismiss a dropdown, etc.
Pressing Escape once may be consumed by the browser. Pressing it again might
exit browse mode, or the browser might consume it again. There is no way to
know.

Ctrl+Esc must ALWAYS exit browse mode with a single keybinding, immediately, no
matter what the browser is doing. This is the keyboard equivalent of clicking
outside the overlay. Without it, the user could get stuck in browse mode with no
keyboard escape route.

Currently Ctrl+Esc does nothing. The `keyCallback` in Surface.zig processes keys
through Ghostty's keybinding system, but there is no binding or special handling
for overlay browse mode.

### Design

**Phase 1: Intercept Ctrl+Esc in `keyCallback` (Surface.zig).**

After key remapping but before keybinding processing (before
`maybeHandleBinding`), check for Ctrl+Esc while the overlay is forwarding:

```zig
// Ctrl+Esc exits browse mode (Issue 607 Experiment 1).
if (event.key == .escape and event.mods.ctrl and event.action == .press) {
    const xpc = @import("apprt/xpc.zig");
    if (xpc.isOverlayForwarding(self)) {
        xpc.notifyNonOverlayClicked(self);
        return .consumed;
    }
}
```

This calls the same `notifyNonOverlayClicked` used when clicking outside the
overlay (Issue 606 Experiment 6). It sets `p.browsing = false`, sends
`mode_changed` to the `web` TUI, and sends `focus_changed(false)` to Chromium.

The check goes before `maybeHandleBinding` so it takes priority over any Ghostty
keybinding that might use Ctrl+Esc. It only fires when `isOverlayForwarding` is
true (pane focused + browse mode), so Ctrl+Esc retains its normal behavior when
not browsing.

### Verification

```bash
cd ghost && zig build
open ghost/zig-out/Ghostty.app --stderr ~/dev/termsurf/logs/ghost.log
cargo run -p web -- https://en.wikipedia.org/wiki/Terminal_emulator
```

Test sequence:

1. Click the overlay to enter browse mode
2. Press Ctrl+Esc — should exit browse mode (web TUI shows control mode)
3. Verify terminal input works normally after exiting
4. Click overlay again to re-enter browse mode
5. Press Ctrl+Esc again — should exit immediately

Pass criteria:

- Ctrl+Esc exits browse mode every time, immediately
- The `web` TUI updates its chrome to show control mode
- Terminal input resumes normally after exiting browse mode
- Ctrl+Esc does nothing when not in browse mode (no crash, no side effects)

### Result: Pass

Ctrl+Esc exits browse mode immediately every time.

### Conclusion

Seven lines in `keyCallback`, before any keybinding or terminal processing:
check for Escape + Ctrl + press while `isOverlayForwarding`, call
`notifyNonOverlayClicked`, return `.consumed`. Reuses the same browse→control
transition as clicking outside the overlay. The escape hatch is now in place
before keyboard forwarding is implemented.

## Experiment 2: Basic key forwarding

### Goal

Type characters in a text field on a Chromium overlay. Press Enter to submit.
Press Backspace to delete. Press Tab to change focus. Arrow keys navigate within
text.

### Chromium branch

Create `146.0.7650.0-issue-607` off `146.0.7650.0-issue-603` (the box demo
branch, which has all mouse/scroll/focus handlers). Add `HandleKeyEvent` and the
`key_event` XPC dispatch on this branch. Update `docs/chromium.md` with the new
branch row.

```bash
cd ~/dev/termsurf/chromium/src
git checkout -b 146.0.7650.0-issue-607 146.0.7650.0-issue-603
```

### Background

The keyboard pipeline needs three pieces:

1. **Ghost (Zig):** Intercept keys in `keyCallback` when `isOverlayForwarding`,
   send them via XPC instead of processing them as terminal input.
2. **XPC message:** Carry the key action, Windows VK code, UTF-8 text, and
   modifier bitmask.
3. **Chromium (C++):** Receive the XPC message, construct a
   `NativeWebKeyboardEvent`, call `ForwardKeyboardEvent`.

Chromium's keyboard event protocol requires two events for a character
keystroke: `kRawKeyDown` (with `windows_key_code`) followed by `kChar` (with
UTF-16 `text`). Non-character keys (arrows, Escape, F-keys) only send
`kRawKeyDown`. Key release sends `kKeyUp`.

Ghostty's `input.Key` enum uses DOM-style names (`.key_a`, `.enter`, `.tab`,
`.arrow_up`). These map to Windows virtual key codes which Chromium uses for
`windows_key_code`. The modifier bitmask is the same as mouse events: shift=1,
ctrl=2, alt=4, meta=8.

The Ctrl+Esc escape hatch (Experiment 1) must remain above the forwarding block
so it always works.

### Design

**Phase 1: Add `keyToWindowsVK` mapping and `sendKeyEvent` to xpc.zig.**

`sendKeyEvent` constructs the XPC message:

```zig
pub fn sendKeyEvent(
    surface: *CoreSurface,

    key: input.Key,
    mods: input.Mods,

```

XPC fields:

| Field              | Type   | Value                                   |
| ------------------ | ------ | --------------------------------------- |
| `action`           | string | `"key_event"`                           |
| `pane_id`          | string | UUID                                    |
| `type`             | string | `"down"`, `"up"`, `"repeat"`            |
| `windows_key_code` | int64  | From `keyToWindowsVK`                   |
| `utf8`             | string | Text generated by keystroke (may be "") |
| `modifiers`        | uint64 | shift=1, ctrl=2, alt=4, meta=8          |

**Phase 2: Forward keys in `keyCallback` (Surface.zig).**

```zig
        xpc.sendKeyEvent(self, event.action, event.key, event.mods, event.utf8);
```

**Phase 3: Add `HandleKeyEvent` to Chromium.**

In `shell_browser_main_parts.h`, add declaration:

```cpp

                    uint64_t modifiers);
```

In `shell_browser_main_parts.cc`, add includes:


Implementation:

```cpp
void ShellBrowserMainParts::HandleKeyEvent(
    const std::string& pane_id, const std::string& type,
    int windows_key_code, const std::string& utf8_text,
    uint64_t modifiers) {
  DCHECK_CURRENTLY_ON(BrowserThread::UI);

  TabState* tab = nullptr;
  for (auto& t : tabs_) {
    if (t->pane_id == pane_id) { tab = t.get(); break; }
  }
  if (!tab) return;

  int web_modifiers = static_cast<int>(modifiers & 0xF);

  auto event_type = blink::WebInputEvent::Type::kRawKeyDown;
  if (type == "up")
    event_type = blink::WebInputEvent::Type::kKeyUp;

  input::NativeWebKeyboardEvent key_event(
      event_type, web_modifiers, base::TimeTicks::Now());
  key_event.windows_key_code = windows_key_code;

  auto* view = tab->shell->web_contents()->GetRenderWidgetHostView();
  if (!view) return;

  view->GetRenderWidgetHost()->ForwardKeyboardEvent(key_event);

  // For key down with text, also send a Char event.
  if (type != "up" && !utf8_text.empty()) {
    input::NativeWebKeyboardEvent char_event(
        blink::WebInputEvent::Type::kChar, web_modifiers,
        base::TimeTicks::Now());
    char_event.windows_key_code = windows_key_code;

    std::u16string text16 = base::UTF8ToUTF16(utf8_text);
    if (!text16.empty()) {
      char_event.text[0] = text16[0];
      char_event.unmodified_text[0] = text16[0];
    }

    view->GetRenderWidgetHost()->ForwardKeyboardEvent(char_event);
  }
}
```

**Phase 4: XPC dispatch in Chromium.**

After the `focus_changed` handler, add:

```cpp
} else if (action && std::string_view(action) == "key_event") {
  const char* pane = xpc_dictionary_get_string(event, "pane_id");
  const char* type_str = xpc_dictionary_get_string(event, "type");
  int vk = (int)xpc_dictionary_get_int64(event, "windows_key_code");
  const char* utf8_raw = xpc_dictionary_get_string(event, "utf8");
  uint64_t mods = xpc_dictionary_get_uint64(event, "modifiers");
  std::string s_pane(pane ? pane : "");
  std::string s_type(type_str ? type_str : "");
  std::string s_utf8(utf8_raw ? utf8_raw : "");
  content::GetUIThreadTaskRunner({})->PostTask(
      FROM_HERE,
      base::BindOnce(&ShellBrowserMainParts::HandleKeyEvent,
                     base::Unretained(self), s_pane, s_type, vk,
                     s_utf8, mods));
}
```

Pass criteria:

- Click the Google search box, type "hello" — characters appear
- Press Backspace — deletes a character
- Press Enter — submits the search
- Press Tab — moves to the next focusable element
- Arrow keys move the cursor within the text field
- Cmd+A selects all text in the field
- Cmd+C copies selected text
- Cmd+V pastes from clipboard
- Ctrl+Esc still exits browse mode (escape hatch preserved)

### Result: Partial pass

Character typing works — clicking the Google search box and typing "hello" makes
characters appear. But pressing Enter freezes the browser overlay. It stops
responding to subsequent key presses and mouse input, then eventually gets
unstuck at a random time later. The freeze is not permanent but makes Enter
unusable.

Backspace, Tab, arrow keys, and Cmd+key shortcuts were not tested because the
Enter freeze blocked further testing on Google.

### Conclusion

The basic key forwarding pipeline works for character input. `keyToWindowsVK`
maps keys correctly, `sendKeyEvent` constructs and sends XPC messages,
`HandleKeyEvent` receives them and calls `ForwardKeyboardEvent`, and typed
characters appear in text fields. The three-layer pipeline (Zig → XPC →
Chromium) is proven end to end.

The initial hypothesis was that Enter specifically caused the freeze, but
further testing revealed the problem is **navigation itself**. Pressing Enter on
Google freezes the overlay. Clicking the Search button (mouse, no keyboard
involved) also freezes it. The same freeze occurs on lite.duckduckgo.com — a
pure HTML page with minimal JavaScript. Wikipedia navigation works. The freeze
is not caused by page weight or JavaScript complexity.

The likely root cause: when a page navigates, Chromium swaps the
`RenderWidgetHostView`. We never re-focus the new view, and the
`FrameSinkVideoCapturer` may lose its capture target. After navigation:

1. The old `RenderWidgetHostView` is destroyed or replaced.
2. We called `Focus()` and `SetActive(true)` on the old view — never on the new
   one.
3. The new view ignores all input because it's not focused/active.
4. The capturer may have lost its `FrameSinkId` target, so no new frames are
   produced and the overlay shows a stale image.

This is a navigation lifecycle problem, not a keyboard problem. It must be
solved before keyboard input can be fully tested. See Issue 608 (search input).

## Conclusion

Issue 607 is suspended. Two experiments were completed:

**Experiment 1 (pass):** Ctrl+Esc escape hatch. Seven lines in `keyCallback`
intercept Ctrl+Esc before any other processing and exit browse mode immediately.
This is the keyboard equivalent of clicking outside the overlay.

**Experiment 2 (partial pass):** Basic key forwarding. The full pipeline works
for character input — typing in text fields produces visible characters. But any
action that triggers page navigation (Enter to submit a form, clicking a Search
button) freezes the overlay. This is not a keyboard-specific bug; it's a missing
navigation lifecycle handler in the Chromium Profile Server.

### What was built

**Ghost (Zig):**

- `keyToWindowsVK` in `xpc.zig` — maps Ghostty's `input.Key` enum to Windows
  virtual key codes (A-Z, 0-9, Enter, Tab, Backspace, Escape, Space, Delete,
  arrows, Home/End, Page Up/Down, Insert, F1-F12, punctuation).
- `sendKeyEvent` in `xpc.zig` — constructs XPC `key_event` message with pane ID,
  action type (down/up/repeat), Windows VK code, UTF-8 text, and modifier
  bitmask. Same pane lookup and message pattern as `sendMouseEvent`.
- Key forwarding block in `Surface.zig` `keyCallback` — after the Ctrl+Esc
  escape hatch, checks `isOverlayForwarding` and routes keys to Chromium via
  `sendKeyEvent` instead of processing them as terminal input.

**Chromium (C++, branch `146.0.7650.0-issue-607`):**

- `HandleKeyEvent` in `shell_browser_main_parts.cc` — receives key action, VK
  code, UTF-8 text, and modifiers. Constructs `NativeWebKeyboardEvent` for
  `kRawKeyDown`/`kKeyUp`, calls `ForwardKeyboardEvent`. For character keystrokes
  (non-empty UTF-8 text), sends a follow-up `kChar` event with UTF-16 text.
- XPC dispatch for `"key_event"` action — extracts fields from XPC dictionary,
  posts `HandleKeyEvent` to the UI thread.

### What blocks completion

Navigation freezes the overlay. When a page navigates (form submission, link
click, button click), the `RenderWidgetHostView` is swapped and we never
re-focus the new view or re-target the video capturer. This affects all input
(mouse and keyboard) and all visual updates. It must be fixed before keyboard
input can be fully validated.
