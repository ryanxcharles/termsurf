# Experiment 1: Restore PDF Embedder CSS Access

## Description

The visible symptom is a full-size webview containing a tiny PDF rectangle near
the top-left. The rectangle looks like a default HTML iframe size, not a zoomed
PDF page.

Chromium's PDF wrapper generated by
`MimeHandlerViewAttachHelper::CreateTemplateMimeHandlerPage()` relies on
`pdf_embedder.css` to size the wrapper iframe:

```css
iframe {
  position: absolute;
  left: 0;
  top: 0;
  height: 100%;
  width: 100%;
  border-style: none;
}
```

Issue 792's TermSurf PDF component manifest snapshot originally included:

```json
"web_accessible_resources": ["pdf_embedder.css"]
```

but `CreateTsPdfComponentExtension()` strips `web_accessible_resources` before
creating the extension. That likely made sense while reducing the PDF extension
surface during plumbing work, but it now explains the tiny iframe: the top-level
PDF wrapper document is not an extension page, so its stylesheet request for
`chrome-extension://mhjfbmdgcfjbbpaeojofohoefgiehjai/pdf_embedder.css` needs the
extension's web-accessible resource policy to allow the load. Without the CSS,
the iframe falls back to browser default dimensions.

This experiment restores access to `pdf_embedder.css` and verifies that the PDF
iframe fills the pane for both the local fixture and `https://bitcoin.org`.

This experiment must receive Claude design review before implementation. After
the result is recorded, Claude must review the completed output before the issue
is closed or the next experiment is designed.

## Pass Criteria

Experiment 1 passes only if:

- Chromium builds;
- the PDF component manifest keeps `pdf_embedder.css` web-accessible;
- logs show `pdf_embedder.css` is requested and served successfully;
- local and remote PDF screenshots show normal full-pane iframe/content sizing;
- local and remote PDFs still render visible PDF content;
- HTML sanity still passes;
- logs do not contradict the result.

## Partial Criteria

Experiment 1 is partial if:

- `pdf_embedder.css` is preserved and served, but the iframe is still tiny;
- the local PDF fixes but the remote PDF does not;
- the iframe becomes full-size but PDF content regresses;
- verification exposes a different downstream sizing gate.

In that case, the next experiment should target the first proven downstream gate
rather than adding zoom or layout guesses.

## Failure Criteria

Experiment 1 fails if:

- Chromium does not build;
- the fix changes PDF zoom instead of iframe sizing;
- the fix touches webtui, Wezboard, Roamium Rust, or the TermSurf protocol;
- the fix regresses Issue 792's inline PDF rendering path;
- the verification uses an installed/stable Roamium instead of the repo-built
  debug binary.

## Conclusion

The hypothesis was correct. The tiny PDF rectangle was not a zoom problem; it
was the PDF wrapper iframe falling back to default iframe dimensions because
`pdf_embedder.css` was no longer web-accessible.

Preserving `web_accessible_resources: ["pdf_embedder.css"]` in the Manifest V2
PDF component extension restored the stylesheet load. The CSS now sizes the PDF
wrapper iframe to the full viewport, and both local and remote Bitcoin PDF URLs
render in the normal full-pane Chrome PDF viewer layout.

Issue 793 can close after Claude reviews the completed experiment and agrees the
result is sound.
